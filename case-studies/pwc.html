PwC DESIGN SYSTEM One Design Language Across a Fragmented Analytics Landscape.

Lead UX Designer — end-to-end

Scale

50–60 dashboards · 7+ business domains

Stakeholders

Internal managers · C-suite · External clients

01Context

EVERYONE WAS BUILDING THEIR OWN VERSION.

PwC's Power BI analytics environment had no design standards. Teams across banking, advisory, insurance, people management, project management, leads, and timesheet functions were each building dashboards independently — off-brand, inconsistent, no shared logic between them.

Over 1.5 years, I led UX for a Power BI design system and 50–60 dashboards across those domains, for stakeholders ranging from internal managers to C-suite executives to external clients at major banks and government agencies.

The Brief

"Design dashboards for our business functions."

Straightforward brief. What sat underneath it was structural: without a design foundation, every new dashboard would inherit the same chaos. This wasn't a dashboard problem. It was a system problem.

02What Was Actually Wrong

THE PROBLEM BEFORE THE DASHBOARD PROBLEM.

PwC had a brand identity. Their Power BI environment ignored it. No palette rules for data visualisation. No standards for which chart type served which decision context. No component library. No layout grid. Each team doing its own thing.

The result: dashboards that looked like they came from different organisations — built for different stakeholders with no shared visual language, requiring users to relearn the interface with every new dashboard they opened.

Fixing individual dashboards one at a time would solve nothing permanently. Every new request would drift again. The design system had to come first.

A DESIGN SYSTEM DOESN'T CONSTRAIN THE WORK. IT'S WHAT MAKES SCALE POSSIBLE WITHOUT LOSING COHERENCE.

Decision 01

Translating Brand Identity into Data Visualisation Rules

PwC's brand guidelines were built for print and digital product — not data visualisation. The translation work was the first real design challenge.

Power BI has hard constraints: limited typography control, native component behaviour, rendering differences across screen sizes. Mapping PwC's colour palette into semantic data viz rules meant deciding which colour signals an alert state, which signals positive trend, which is neutral — and making those rules consistent across banking, insurance, and HR data models that have very different threshold logic.

The constraint that forced discipline: once a colour carries meaning in one dashboard, it has to carry the same meaning in all of them. That's what makes a system rather than a collection of dashboards.

Decision 02

Chart Type as a Decision Rule, Not an Aesthetic Choice

The question isn't "what chart looks good here?" It's "what decision does this data support, and which chart makes that decision easiest?"

Standardising chart type selection across domains required understanding decision-making patterns, not just data types. Time-series for trend analysis and forecasting quality. Bar for cross-entity comparison. Donut for composition at a glance. KPI cards for trigger-level signals that need to be read before everything else.

The same rules applied to a banking performance dashboard and a people management dashboard — different data, same decision logic. Consistency here is what trains the user to read faster over time.

Decision 03

Stakeholder-Adaptive Layouts Within a Single System

A dashboard for an internal delivery manager and one for an external banking client have fundamentally different cognitive requirements — different information hierarchies, different levels of detail, different tolerances for ambiguity. A government agency client needs different framing than a C-suite executive reviewing the same underlying data.

The design system had to be specific enough to enforce visual consistency and flexible enough to serve all these contexts without breaking into parallel systems. The layout grid and component patterns were built to flex for stakeholder type — density, hierarchy depth, labelling detail — while the brand language stayed fixed.

Decision 04

Components That Reduced Design Debt on Every Subsequent Dashboard

KPI cards, filter panels, chart containers, navigation — standardised as reusable components. Any new dashboard request could be composed from the library rather than starting from scratch, which meant every subsequent dashboard was faster to build and automatically on-brand.

This was the compounding return on the upfront system work: the 50th dashboard took less design effort than the 5th, and looked more consistent than if each had been designed independently.

05Learnings

DESIGN SYSTEMS ARE NOT DOCUMENTATION. THEY'RE DECISIONS.

The hardest part wasn't defining the components. It was getting the chart type standards right — because that required understanding decision-making patterns across radically different business domains, not just visual design principles. A chart that works for banking performance data doesn't automatically work for people management analytics.

The other thing this project reinforced: communicating the reasoning behind every system decision to stakeholders who had never thought about their dashboards as a system before is itself a design challenge. The system only holds if the people using it understand why the rules exist.

BEFORE YOU CAN DESIGN 50 DASHBOARDS, YOU HAVE TO DECIDE WHAT MAKES A DASHBOARD.
