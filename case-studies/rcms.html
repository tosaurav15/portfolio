01Context

ONE DELIVERY ORG. TWO DIFFERENT CONTRACTS.

HPE's global delivery organisation runs projects under two fundamentally different commercial models. Staff Augmentation: NCG consultants embedded in HPE projects, billed by time and rate. Managed Services: NCG delivers a defined scope, billed against milestones and cost targets.

Same delivery org. Same roles. Completely different financial logic, approval responsibilities, and billing triggers. Before RCMS, this was managed through a patchwork of spreadsheets, email threads, and disconnected tools — one for timesheets, another for approvals, nothing for billing visibility.

The result: fragmented data, approval delays, billing discrepancies, and no single source of truth for project status or resource cost across the org.

The Brief

"Build a unified platform for HPE's delivery org — covering resource management, project tracking, timesheets, multi-role approvals, and billing — across both Staff Augmentation and Managed Services engagements."

The brief described a feature list. The design problem underneath it was architectural — and considerably harder.

02The Real Problem

SAME ROLE. DIFFERENT QUESTION.

The brief asked for "one unified platform." The problem was that the two engagement models aren't just operationally different — they answer different questions at the same workflow step.

A Delivery Manager approving a Managed Services cost is asking: did we deliver the scope against budget? The same DM approving a Staff Augmentation timesheet is asking: did this resource log accurate hours against allocation? Same role, same approval module, completely different decision context.

BUILD ONE LAYOUT FOR BOTH MODELS AND THE SYSTEM FORCES APPROVERS TO RECONSTRUCT CONTEXT BEFORE EVERY DECISION. THAT'S NOT A UI PROBLEM — IT'S A STRUCTURAL ONE.

The design challenge: build a platform coherent enough that it feels like one system, while making the bifurcation explicit enough that users are never confused about which model they're operating in. Invisible complexity causes errors. Over-surfaced complexity causes fatigue.

RCMS had to thread that needle — across 8 modules, 6 user roles, and a financial lifecycle that runs from rate cards through timesheets to monthly invoices.

03Design Decisions

FOUR DECISIONS THAT SHAPED THE SYSTEM.

Decision 01

Calendar Timesheet — Zero Learning Curve by Design

Every enterprise tool defaults to forms. A timesheet form asks: date, project, hours, activity. The user has to translate their week into fields. A calendar model removes the translation entirely — you log time where time lives.

My team's default was a form. I pushed back and proposed a calendar interaction model — and had to fight for it internally. The argument: every person in HPE's delivery org already operates in Google Calendar or Outlook daily. The calendar mental model — time mapped to days, weeks as units, visual density showing load — already exists in every user's head before they open RCMS.

For a global deployment of 2,000 users, eliminating the learning curve isn't a UX nicety. It's adoption at scale. No training programme was needed for timesheet onboarding. Users opened the calendar and understood immediately — because they'd been using this interaction model in their daily tools for years.

The calendar also surfaced patterns that a form couldn't: visual load distribution across the week, project overlap, gaps that could prompt before they became compliance issues.

Decision 02

Two-Model Approval Architecture — Context Before Data

HPE had the approval hierarchy. What wasn't working was that every approver had to mentally reconstruct context before they could act. The fix: establish model context at the point of entry — before the data is shown.

The three approver roles — Delivery Manager, PMO, and Requester PM — each had defined authority. The architecture was HPE's. What we redesigned was the flow and the data interaction model.

Staff Augmentation and Managed Services generate different approval views for the same role. A DM viewing Staff Aug is checking timesheet actuals against resource allocation. The same DM viewing Managed Services is checking cost actuals against scope and budget. The underlying question is different. The data layout needs to reflect that difference — clearly, before the approver touches anything.

The redesign: model-specific view architectures within the same approval module. Navigation into an approval establishes the frame (which model, which type). The layout answers the right question for that model. Reconciliation views — showing discrepancies between actuals and approved figures — are a cross-check surface, not a duplicate workflow.

Decision 03

Rate Card → Billing Through-Line — Making the Financial Chain Visible

The rate card model was HPE's. The problem was that a rate card error doesn't surface as an error at the Admin module — it surfaces three steps later as a billing discrepancy, by which point the cause is obscured and the fix is disruptive.

I designed the interaction between the Admin (rate cards), Projects (resource allocation), and Billing modules to make the causal chain visible at each step. Assigning a rate card to a resource shows the downstream billing implication before confirmation. The billing calendar surfaces the rate card source for each line item — errors are traceable, not just visible. Monthly lock — the step that closes a billing period — is a deliberate confirmation surface with downstream consequences made explicit.

The goal: a PMO or finance user moving through rate cards, allocation, and billing shouldn't need to hold the connection between these modules in their head. The system makes it visible at the moment it matters.

Decision 04

Dark Mode Parity — A Design System Decision, Not a Feature

Dark mode wasn't a HPE requirement. It was my call. Delivery Managers and PMO users work long hours, often across time zones. A system that fatigues the people using it most intensively has a design problem.

Building dark mode parity meant designing every approval view, every timesheet screen, and every profile surface twice — not as an afterthought, but as a design system constraint from the start. Light and dark aren't two colour themes. They're two rendering contexts that both have to work at equal fidelity.

The constraint forced better decisions on contrast, hierarchy, and colour usage than a light-only design would have. When a component has to work on both surfaces, you can't lean on background colour to do structural work. The hierarchy has to be explicit in the layout, not just the palette.

Screen — Calendar Timesheet, Dark Mode

Full Parity

Every element present in light mode is present in dark — WBS panel, calendar grid, planned vs actual pills, legends, CTAs. Not a dark colour wash. Full re-considered rendering.

User-Controlled Switch

Light / Dark toggle visible in the top right — the user controls their environment. For a system used across time zones and long sessions, this is not optional.

Layout-Led Hierarchy

Colour-coded pills maintain their meaning in dark mode — approval status reads the same way regardless of theme. Hierarchy lives in the layout, not the background.

04Project Lifecycle

METHODOLOGY AS NAVIGABLE STRUCTURE.

HPE runs projects through a defined delivery framework: Delivery Initiation → Start Right → Execute Right → Close Right → Compliance. This methodology was HPE's. Translating it into UX was the design challenge.

The risk with a phased methodology inside a product is that phases become tabs — passive labels that show status but don't drive action. A Project Manager opens their project, sees "Execute Right," and still has to know from memory what that phase requires.

A PHASE ISN'T A LABEL. IT'S A SURFACE THAT SHOWS WHAT'S DONE, WHAT'S PENDING, AND WHAT'S REQUIRED BEFORE THE NEXT PHASE OPENS.

Each phase in the Projects module is an active surface. Document requirements, scope items, resource plans — surfaced within the phase they belong to, at the moment they're relevant. MoU generation, scope documents, compliance checks: not separate features buried in a documents section. Part of the phase they gate.

The result: the delivery methodology becomes structure the system enforces, not a framework the user has to remember.

06Learnings

STRUCTURE THAT HOLDS.

The hardest design problem in RCMS wasn't any individual module. It was maintaining coherence across a system with genuine architectural branching at its core. Two engagement models that are structurally different — but have to feel like one product.

The lesson: when a system has real bifurcation, the design response isn't to hide it. It's to make the bifurcation explicit at precisely the moment it matters — and invisible everywhere else. The model context needs to be established before the user touches data. After that, it can recede.

A SYSTEM DOESN'T FAIL WHERE IT ACTS. IT FAILS WHERE IT ASSUMES THE USER ALREADY KNOWS WHICH CONTEXT THEY'RE IN.

The rate card → billing intelligence layer — surfacing anomalies upstream before they become billing errors — is the obvious next design problem. That's the version of this system that would make the financial chain not just traceable but proactive.
